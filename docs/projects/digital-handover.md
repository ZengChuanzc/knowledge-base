# 数字化移交通道管理平台

> 面向大型集团工程全生命周期移交场景，搭建**外网业务通道 + 内网生产环境**双模块分布式管理平台。解决建设、施工、监理、管理等多单位协同效率低、数据不规范、内外网数据隔离、流程审核不透明等行业痛点。

## 二、AI Code Review 自动审查组件

### 1. 业务背景与痛点

大型集团项目涉及多个开发团队并行开发，**代码审查质量参差不齐**，完全依赖人工审查导致：

| 问题 | 影响 |
|------|------|
| 审查遗漏 | 人工审查易漏掉安全漏洞和 SQL 注入风险 |
| 标准不统一 | 每个审查者的标准不同，同一类问题有时报有时不报 |
| 效率低下 | MR 等待人工审查，阻塞合入流程 |
| 重复劳动 | 相同类型的问题反复被提出，没有形成知识积累 |

### 2. 方案选型对比

| 方案 | 描述 | 优点 | 缺点 | 结论 |
|------|------|------|------|------|
| **方案A：纯静态规则（SonarQube + PMD）** | 预定义规则集扫描 | 稳定可靠，无幻觉 | 只能检测模式化问题，无法理解业务语义 | ❌ 不够 |
| **方案B：LLM 审查** | 调用大模型分析代码 Diff | 能理解语义，发现逻辑漏洞 | 有误报，需要 Prompt 调优 | ✅ **选择** |
| **方案C：LLM + RAG 增强** | LLM 审查 + 向量检索历史案例 | 结合语义理解和历史经验 | 实现复杂度高 | ✅ **组合方案B** |

**为什么不用纯规则引擎？** 项目需要检测的不仅是代码规范问题，还有业务逻辑漏洞（如事务失效、权限绕过），这些纯规则引擎无法理解。LLM 能够结合上下文分析，但会引入误报问题——这需要通过 Prompt 工程来解决。

### 3. 技术架构

```
GitLab MR 创建/更新
        │
        ▼
   GitLab Webhook ───→ Code Review Service
        │                      │
        │               ┌──────┴──────┐
        │               │  提取增量   │
        │               │  Diff 内容  │
        │               └──────┬──────┘
        │                      │
        │               ┌──────┴──────┐
        │               │  LLM 分析   │
        │               │  Qwen 模型  │
        │               │  私有化部署  │
        │               └──────┬──────┘
        │                      │
        │               ┌──────┴──────┐
        │               │  Prompt 优化 │
        │               │  ├─ Few-shot│
        │               │  └─ 否定指令│
        │               └──────┬──────┘
        │                      │
        ▼                      ▼
   MR 评论区 ←───────── 审查结果回写
```

### 4. 误报率优化过程

```
初始状态（基础 Prompt）：        误报率 22%
    ↓
添加 Few-shot 正反示例：         误报率 15%
    ↓
添加否定指令（明确不报的场景）：   误报率 9% ✅
    ↓
添加 Embedding 历史案例检索：    误报率 9%（稳定性提升）
```

#### 为什么 Few-shot 有效？

LLM 在没有示例的情况下，倾向于"过度审查"——看到一点可疑就报，宁可错杀不可放过。给了正反示例后，模型知道"什么算问题、什么不算"，判断更准确。

```java
// Few-shot 示例
examples = [
    { bad: "@Transactional\nprivate void save() {}", good: "@Transactional\npublic void save() {}" },
    { bad: "String sql = \"SELECT * FROM user WHERE id=\" + id", good: "String sql = \"SELECT * FROM user WHERE id=#{id}\"" },
]
```

#### 为什么否定指令进一步降低误报？

即使有 Few-shot，模型在某些边缘场景仍会误报。否定指令明确划出边界：

```java
String negativeInstruction = """
    以下情况**不要**报错：
    1. 日志输出、注释变更不属于代码问题
    2. 测试代码中的 Mock 数据不属于安全问题
    3. 符合项目命名规范的变更不属于规范问题
    4. 仅仅是格式化或缩进调整不属于代码问题
    """;
```

#### 收益

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 误报率 | **22%** | **9%**（↓59%） |
| 审查效率 | 人工逐行审查 | 30s 内自动返回结果 |
| 审查一致性 | 依赖审查者经验 | 统一标准，输出稳定 |
| 知识积累 | 靠个人记忆 | Embedding 持续沉淀历史案例 |

## 三、多节点审核流程

### 1. 业务背景与痛点

工程移交涉及**建设单位、施工单位、监理单位、管理机构**等多方协同，原来审核流程是**线下流转**：

```
线下流程：
  提交纸质资料 → 电话通知监理 → 监理签字 →
  物理送到建设单位 → 建设单位签字 → 归档
  （平均周期：3-7 天）
```

| 问题 | 影响 |
|------|------|
| 效率低 | 线下传递资料、电话催办，周期长 |
| 不透明 | 审批卡在谁手里不知道，只能挨个问 |
| 难追溯 | 签字纸质存档，查找困难 |

### 2. 方案对比

| 方案 | 做法 | 优点 | 缺点 | 结论 |
|------|------|------|------|------|
| **方案A：简单工单系统** | 一个流程走到底，无分支 | 实现简单 | 不支持多单位会签、驳回、条件分支 | ❌ |
| **方案B：BPMS 流程引擎** | 图形化配置流程节点，支持会签/驳回/条件路由 | 灵活强大，适配复杂业务 | 需要一定的配置学习成本 | ✅ **选择** |
| **方案C：自研状态机** | 代码定义状态流转 | 自由度高 | 流程变更要改代码，维护成本高 | ❌ |

**为什么选 BPMS？** 移交通道的审核流程不是固定的——不同类型的移交（设计移交、施工移交、竣工移交）有不同的审核节点和审批人。BPMS 支持**可视化流程配置**，业务变更时不需要改代码，运营人员自己配置即可。

### 3. 实现方案

```yaml
# BPMS 流程定义示例（竣工移交审核流程）
流程节点：
  - 步骤1: 提交竣工资料         → 角色: 施工单位
  - 步骤2: 监理审核             → 角色: 监理单位
            ├── 通过 → 下一步
            └── 驳回 → 退回步骤1（附驳回意见）
  - 步骤3: 建设单位审批         → 角色: 建设单位
            ├── 通过 → 下一步
            ├── 驳回 → 退回步骤2
            └── 会签 → 多部门同时审批
  - 步骤4: 归档                 → 角色: 系统自动
```

### 4. 收益

| 指标 | 优化前（线下） | 优化后（线上 BPMS） |
|------|--------------|-------------------|
| 审核周期 | 3-7 天 | **缩短 40%** |
| 流程透明 | ❌ 不透明 | ✅ 全程可追踪 |
| 协同方式 | 电话+纸质 | 系统自动通知+线上办理 |
| 人力成本 | 需专人催办 | 超时自动催办 |

## 四、思考：是否还有更合适的方案

### 4.1 AI Code Review，能不能用 GPT 替代 Qwen？

项目选择了私有化部署 Qwen 而不是调用 GPT API，核心原因是**数据安全**——项目涉及集团内部工程资料和代码，不能出内网。Qwen 私有化部署虽然模型能力略逊于 GPT-4，但在安全合规上满足要求。

如果场景是开源项目或非敏感代码，GPT-4 的审查质量更高，而且不需要维护本地推理资源。**选择 Qwen 是安全约束下的最优解，不是技术上的最优解。**

### 4.2 误报降到 9% 够不够？能不能更低？

9% 的误报率意味着每 100 条审查意见中有 9 条是误报。理论上可以更低：

| 手段 | 预期效果 | 成本 |
|------|---------|------|
| 更多高质量的 Few-shot 示例 | 可能降到 5% | 需要人工标注大量正反例 |
| 更细致的否定指令 | 微降 | 持续迭代 Prompt |
| 人工反馈闭环（标记误报→模型学习） | 持续下降 | 需要建立反馈机制 |

但实际中 9% 的误报率在可接受范围内——研发看到 10 条意见里有 1 条误报，并不会影响他们对这个工具的信任。如果把精力放在追求 0 误报上，边际效益很低。

### 4.3 BPMS 审核流程，有没有更轻量的方案？

BPMS 功能强大但相对重，如果只是简单的审批流（没有会签、驳回、多分支），用**状态机 + 配置表**也能实现：

```
数据库表：
  approval_node (id, step, role, next_on_pass, next_on_reject)
  
代码逻辑：
  // 根据当前节点和审核结果，查询配置表决定下一步
  NodeConfig config = nodeConfigMapper.getNextNode(currentNode, result);
```

这样更轻量，不需要引入 BPMS 引擎。但项目中选择 BPMS 是预见到业务会越来越复杂，需要灵活配置。**如果回到需求初期能确认流程不会太复杂，状态机方案更轻便。**

### 4.4 内外网双模块，有没有更好的数据交换方式？

当前是外网提交申请 → 安全隔离区 → 内网审核处理。替代方案：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **API 网关 + 白名单** | 架构统一 | 安全等级不够 |
| **现有隔离区方案** | 安全可控 | 额外维护隔离区组件 |
| **消息队列跨网桥接** | 异步解耦 | 复杂度最高 |

现有方案在安全性和实现复杂度之间取得了平衡。如果要更严格的安全等级，可以采用**物理隔离 + 摆渡机/U盘**方式，但那样实时性会更差。**
